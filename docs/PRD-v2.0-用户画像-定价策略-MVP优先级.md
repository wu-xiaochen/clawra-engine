# 一、目标用户画像 (User Persona)

## 1.1 核心用户群体

### Primary User - 技术型采购工程师

| 维度 | 描述 |
|------|------|
| **角色** | 采购工程师 / 供应链专员 |
| **年龄** | 28-45岁 |
| **学历** | 本科及以上（工科背景优先） |
| **企业规模** | 中小型企业（50-500人） |
| **年采购金额** | 50万-500万人民币 |
| **采购品类** | 工业设备、原材料、零部件 |

**核心痛点**:
- 面对复杂技术参数无从下手，选型依赖供应商推荐
- 需要向领导证明决策合理性，但缺乏专业依据
- 缺乏供应链风险意识，被动应对断供、涨价

**使用场景**:
- 日常采购选型（每周3-5次）
- 供应商年度审核
- 新项目设备选型

**决策因素优先级**:
1. 决策依据充分性（能否说服领导）
2. 效率提升（能否快速完成选型）
3. 风险可控性（是否降低供应链风险）

---

### Secondary User - 研发/技术人员

| 维度 | 描述 |
|------|------|
| **角色** | 研发工程师 / 技术支持 |
| **年龄** | 25-40岁 |
| **企业规模** | 中小型企业 |
| **职责** | 提供技术选型建议、参与采购评审 |

**核心痛点**:
- 被频繁咨询技术参数问题，耗时耗力
- 供应商技术文档不完整，难以快速评估
- 希望积累技术知识库，减少重复咨询

**使用场景**:
- 为采购提供技术选型建议
- 评估供应商技术能力
- 建立企业内部技术知识库

---

### Tertiary User - 管理层/决策者

| 维度 | 描述 |
|------|------|
| **角色** | 采购主管 / 供应链经理 / 总经理 |
| **年龄** | 35-55岁 |
| **企业规模** | 中小型企业 |
| **职责** | 审批采购决策、管理供应链风险 |

**核心痛点**:
- 采购决策缺乏数据支撑，风险不可知
- 供应商信息不透明，难以横向对比
- 希望看到采购成本趋势和风险全景

**使用场景**:
- 审批重大采购项目
- 供应商战略评估
- 供应链风险管理

# 二、定价策略 (Pricing Strategy)

## 2.1 定价模型设计

### 定价原则

1. **价值导向**: 基于用户节省的时间成本和降低的风险定价
2. **分层定价**: 满足不同规模企业的需求
3. **免费获客**: 提供免费版用于市场教育
4. **年付优惠**: 鼓励长期合作，提高LTV

---

### 定价层级

| 特性 | 免费版 (Free) | 专业版 (Pro) | 企业版 (Enterprise) |
|------|---------------|--------------|---------------------|
| **目标用户** | 个人试用 |  SMB团队 | 中大型企业 |
| **价格** | ¥0 | ¥99/月 | 定制化报价 |
| **年付价格** | - | ¥990/年 (省18%) | 定制 |
| **本体库** | 基础品类(1个) | 全品类(10个) | 无限定制 |
| **推理次数/月** | 50次 | 1,000次 | 无限 |
| **供应商评估** | 基础5家 | 无限 | 无限 |
| **知识图谱** | 100条实体 | 10,000条 | 无限 |
| **API调用** | ❌ | ✅ | ✅ |
| **专属客服** | ❌ | ❌ | ✅ |
| **定制本体** | ❌ | ❌ | ✅ |
| **私有部署** | ❌ | ❌ | ✅ |

---

### 2.2 定价策略说明

#### 免费版定位
- **目的**: 市场教育 + 产品试用
- **策略**: 限制推理次数和功能深度，确保核心价值可体验
- **转化目标**: 试用用户中10%转化为付费

#### 专业版定位
- **目的**: 主力变现产品
- **定价依据**:
  - 采购工程师月薪约8,000-15,000元
  - 使用产品后每周节省5小时*4周=20小时/月
  - 时薪价值约50元，则月节省1,000元
  - ¥99/月定价为节省成本的10%，性价比极高
- **策略**: 强调"一次选型失误的损失 = 1年会员费"

#### 企业版定位
- **目的**: 高客单价 + 深度绑定
- **定价**: 年采购金额的0.5%-1%作为参考基准
- **策略**: 提供私有部署、数据本地化、定制开发

# 三、MVP功能优先级 (Feature Prioritization)

## 3.1 功能分级框架

基于 **用户价值** 和 **技术实现难度** 双重维度评估：

```
价值高 + 实现简单 → P0 立即做
价值高 + 实现复杂 → P1 规划做  
价值低 + 实现简单 → P2 可以做
价值低 + 实现复杂 → P3 后续做
```
